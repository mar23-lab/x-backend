**Companion:** [`CLERK_OAUTH_PROVIDER_CONFIG.md`](./CLERK_OAUTH_PROVIDER_CONFIG.md), R50.3d Slice 1 (PR #409 · `src/workers/routes/sources.ts` `POST /sources/:id/sync`).

---

## 1. Goal

R50.3d Slice 1 (PR #409) wired the **manual** path: `POST /api/v1/sources/:id/sync` verifies the
user's Clerk OAuth token and dispatches the per-provider translator (`getTranslator(provider)` →
`runTranslator(TranslatorInput)`), which pulls metadata-only events into the operator's event store
tagged `source_type = 'live_api'`.

Slice 2 adds the **scheduled** path: a background loop that periodically syncs every *connected*
source without a user request, so the cockpit stays fresh between manual syncs.

This document is the complete implementation blueprint. It is intentionally **not implemented yet** —
§6 explains why this is the correct call, and §5 is the exact step-list to implement it safely once a
live connection exists.

---

## 2. The binding constraint — wrangler is at the 5-cron account cap

Cloudflare Workers (this account's plan) enforces **max 5 cron triggers per worker** (deploy error
`code: 10072 · limit of 5 cron triggers`). `wrangler.toml [triggers] crons` already lists **5** (all used):

| # | Expression | Wrangler-fired loop |
|---|---|---|
| 1 | `*/5 * * * *` | propagation_tick |
| 2 | `0 * * * *` | permanent_suppress (hourly) |
| 3 | `0 4 * * *` | threshold_retune |
| 4 | `0 5 * * *` | calibration_retrain |
| 5 | `0 3 * * 1` | weight_retune (weekly) |

**A 6th wrangler trigger is impossible without a paid-plan upgrade.** Therefore source-sync must run by
**chaining onto an existing wrangler-fired handler** — exactly the pattern already used in-repo.

### 2.1 The established chain pattern (verified 2026-06-05)

The `scheduledHandler` dispatcher (`src/workers/index.ts:197`) looks up **one** handler per cron
expression: `const entry = CRON_BY_EXPRESSION[event.cron]`. So loops that don't own a wrangler trigger
run by being **called at the tail of a sibling that does**:

- `src/workers/crons/threshold-retune.ts:48` → `const chained = await patternSuspendCron(ctx);`
  (threshold_retune fires `0 4 * * *`; pattern_suspend has no wrangler trigger of its own)
- `src/workers/crons/calibration-retrain.ts:53` → `const chained = await shadowEvalCron(ctx);`
  (calibration_retrain fires `0 5 * * *`; shadow_eval has no wrangler trigger of its own)

The chained handler's result is surfaced in the parent's `metadata.chained` for log visibility; the
dispatcher's `try/catch` (`index.ts:207-218`) means a chained throw is logged, not fatal to the parent.

> **⚠️ Latent doc-accuracy finding (record + fix separately).** `CRON_REGISTRY`
> (`src/workers/crons/index.ts`) lists `pattern_suspend` with `cron: '30 4 * * *'` and `shadow_eval`
> with `cron: '15 5 * * *'`, and `index.ts:188-190` comments them as if they were live triggers. **They
> are NOT in `wrangler.toml`**, so Cloudflare never sends those expressions — those two loops run *only*
> via the chains above. The registry `cron` field for chained loops is documentation of intended
> cadence, not a real trigger. Source-sync should follow the same convention but the misleading comments
> should be corrected to say "chained via &lt;parent&gt; — not a wrangler trigger."

### 2.2 Cadence decision → chain onto the **hourly** `permanent_suppress`

| Candidate parent | Cadence | Verdict |
|---|---|---|
| propagation_tick | every 5 min | ✗ too frequent — external-API rate-limit + Cloudflare CPU + cost risk |
| **permanent_suppress** | **hourly** | ✓ right cadence for external source polling; lowest-traffic hot path |
| threshold/calibration | daily | ✗ too stale for "live" cockpit data |

**Recommendation:** chain `sourceSyncCron` at the tail of `permanentSuppressCron` (`0 * * * *`).

## 3. The second constraint — the cron context has no `env`

`CronHandlerContext` (`src/workers/crons/types.ts`) is `{ dal: DalAdapter, now: () => Date, cronExpression: string }`.
It deliberately has **no `env`** so handlers are unit-testable in isolation.

But source-sync needs a Clerk OAuth token per user, and the only way to get one is
`makeClerkOAuthAdapter(env.CLERK_SECRET_KEY).getAccessToken(...)` — which needs the secret from `env`.
The `/sources/:id/sync` route does exactly this (`src/workers/routes/sources.ts:217-228`).

**The dispatcher has `env`** (`scheduledHandler(event, env, _ctx)` · `index.ts:199`; it already builds
`dal` from `env.DATABASE_URL`). So the fix is: the dispatcher constructs the OAuth adapter from `env`
and **injects it into the cron context**. This is a shared-type change touching all 7 crons (additive,
optional field), which is precisely why it belongs in a reviewed change, not an unattended overnight one.

## 4. Concrete implementation sketch

### 4.1 DAL method (ripples to exactly 2 files — `DalAdapter` has one implementer: `WorkersDalAdapter`)

```ts
// src/workers/dal/DalAdapter.ts  (interface)
/** R50.3d Slice 2 · list every CONNECTED user source for the scheduled sync loop. */
listActiveUserSourcesForSync(limit?: number): Promise<UserSourceConnection[]>;

// src/workers/dal/WorkersDalAdapter.ts  (impl — mirror getUserSource's column list + rowToUserSourceConnection)
async listActiveUserSourcesForSync(limit = 100): Promise<UserSourceConnection[]> {
  const rows = await this.sql`
    SELECT <same columns as getUserSource>
      FROM user_source_connections
     WHERE status = 'connected'            -- status lifecycle: 'pending_auth' -> 'connected' (see createUserSource ~line 623)
     ORDER BY last_sync_at ASC NULLS FIRST -- oldest / never-synced first
     LIMIT ${limit}
  `;
  return rows.map(rowToUserSourceConnection);
}
```
`status = 'connected'` is the gate: until a live Clerk OAuth connection exists, this returns `[]` and the
loop is a no-op (natural dormancy — same property that makes Slice 1 safe to have already merged).

### 4.2 Context extension

```ts
// src/workers/crons/types.ts
import type { ClerkOAuthAdapter } from '../dal/clerk-oauth-adapter';
export interface CronHandlerContext {
  readonly dal: DalAdapter;
  readonly now: () => Date;
  readonly cronExpression: string;
  readonly oauthAdapter?: ClerkOAuthAdapter;  // R50.3d Slice 2 · injected by dispatcher for source_sync
}
```

### 4.3 The handler (new file `src/workers/crons/source-sync.ts`)

```ts
import type { CronHandler } from './types';
import { getTranslator } from '../sources/translators';

const THIRTY_DAYS_MS = 30 * 24 * 60 * 60 * 1000;

export const sourceSyncCron: CronHandler = async (ctx) => {
  const adapter = ctx.oauthAdapter;
  if (!adapter) return { ok: false, metadata: { skipped: 'no_oauth_adapter_in_context' } };

  const sources = await ctx.dal.listActiveUserSourcesForSync();
  let synced = 0, failed = 0, skipped = 0;

  for (const src of sources) {
    const translator = getTranslator(src.provider);
    if (!translator) { skipped++; continue; }
    try {
      await adapter.getAccessToken(src.user_id, src.provider, { force_refresh: true }); // throws if revoked
      const result = await translator({
        adapter, dal: ctx.dal, userSource: src,
        since: src.last_sync_at ?? new Date(ctx.now().getTime() - THIRTY_DAYS_MS).toISOString(),
        max_events: 200,                       // per-tick cap — bounds cost/rate per provider
      });
      await ctx.dal.markUserSourceSync(src.user_id, src.id, { success: result.errors.length === 0 });
      synced++;
    } catch (e: any) {
      await ctx.dal.markUserSourceSync(src.user_id, src.id, { success: false, error: e?.message ?? String(e) });
      failed++;
    }
  }
  return { ok: true, metadata: { total: sources.length, synced, failed, skipped } };
};
```

### 4.4 Wiring (the reviewed hot-path edits — deliberately deferred)

1. `src/workers/index.ts` scheduledHandler: build `const oauthAdapter = makeClerkOAuthAdapter(env.CLERK_SECRET_KEY)`
   and pass it in the ctx object (harmless to the other 6 loops; only source_sync reads it).
2. `src/workers/crons/permanent-suppress.ts`: at the tail, mirror threshold-retune:48 —
   `const chained = await sourceSyncCron(ctx);` then surface `metadata.chained = chained`.
3. `src/workers/crons/index.ts`: `import { sourceSyncCron }`, `export { sourceSyncCron }`, and add a
   `CRON_REGISTRY` entry `{ cron: '0 * * * *', loop_name: 'source_sync', handler: sourceSyncCron, description: 'R50.3d Slice 2 · chained via permanent_suppress — NOT a standalone wrangler trigger' }` for admin-dispatch visibility. **Do not** add a 6th expression to `wrangler.toml`.
4. **Kill-switch:** gate the chain on `env.SOURCE_SYNC_CRON_ENABLED === 'true'` (a `[vars]` flag, default unset/false) so it can be disabled without a redeploy.

## 5. Live-verification plan (run in this order — none is possible until a connection exists)

1. Operator completes the two open OAuth items (Dropbox `files.metadata.read` scope; enable Google Drive API) — see `CLERK_OAUTH_PROVIDER_CONFIG.md` 2026-06-05 update.
2. Connect one account (GitHub is simplest) via the cockpit `SourceConnectorModal` → row `status` flips `pending_auth` → `connected`.
3. **Manual first** (Slice 1, already shipped): `POST /api/v1/sources/:id/sync` → confirm real events land tagged `source_type='live_api'` and `last_sync_at` updates. This proves the translator path end-to-end before any automation.
4. Only then implement §4 + §5-kill-switch, deploy, and watch the first hourly tick: log line `[cron:permanent_suppress] {"metadata":{"chained":{"synced":1,...}}}`.
5. Confirm `markUserSourceSync` advanced `last_sync_at` and a second tick does an incremental (`since = last_sync_at`) pull, not a full re-pull.

**Rollback:** flip `SOURCE_SYNC_CRON_ENABLED=false` (instant, no deploy) or remove the chain line in
`permanent-suppress.ts` and redeploy. No schema or wrangler-trigger changes to revert.

## 6. Why design-only tonight (risk rationale)

Shipping this unattended overnight was rejected for four compounding reasons:

1. **It modifies a production hot path.** `permanent_suppress` runs hourly in prod; a defect in the
   chained loop executes every hour unattended.
2. **Untestable SQL on a 4,217-LoC god-file DAL.** `listActiveUserSourcesForSync` cannot be run without a
   live Neon DB; typecheck would pass while the SQL stays unproven (the "complete-but-shallow" trap).
3. **A shared-context shape change** (`oauthAdapter` on `CronHandlerContext`) touches all 7 crons and the
   dispatcher — a reviewed change, not a 3am one.
4. **No way to verify the happy path.** With zero connected sources, an end-to-end test is impossible; the
   loop is inert until the operator finishes the OAuth setup. There is nothing to gain by shipping it
   before step 5.1 is done, and real downside (latent breakage in a verified-looking PR).

The honest overnight deliverable is therefore this blueprint plus the verified facts it rests on. When a
live connection exists, §4–§5 is a single reviewed sitting.
