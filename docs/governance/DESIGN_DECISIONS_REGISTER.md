# Design Decisions Register — the unified cross-seat SSOT

**Why this exists (F14, 260708):** the backend seat built two surfaces (a billing upgrade-request WRITE and
runtime-policy enforcement) that the design prototype had already RATIFIED AGAINST (§135 read-only billing,
§146 P2 hold), because the ratified decisions lived only inside an 800KB design HTML and were never consulted
before the scope-fork was put to the operator. The operator kept both (his override right) and the design seat
then accepted the divergence — but the process was wrong. This register is the greppable, in-repo SSOT for
cross-seat product/design decisions that BIND backend scope.

**THE CONSULT-FIRST RULE (binding on every agent in this repo):** before building any customer-facing
surface, or presenting any scope-fork (build-X-vs-Y) to the operator, CHECK this register AND the design
plan's ratified sections (`Xlooop Implementation Plan.html`, §-anchored — grep for "ratified", "hold",
"do not build", "decided", "read-only"). If a fork is already decided, surface THAT — "the prototype ratified
X at §N; override?" — never a neutral A/B. Honest limit: this rule is process, not a CI gate; the register
exists so the check is a 10-second grep instead of an 800KB read.

Status key: **BINDING** (constrains scope now) · **RULING** (design-seat call, adopted) · **ACCEPTED-DIVERGENCE**
(built contrary to a ratified decision; operator override on record; design seat accepted).
